❯ now that i upgraded my pip, will it also upgrade my pip in my pc?             

● It depends on where you ran the upgrade command:

  - If you upgraded pip inside a virtual environment (e.g., .venv, conda env):
  No, it only affects that environment. Your system pip remains unchanged.
  - If you upgraded pip globally (e.g., pip install --upgrade pip outside any
  venv): Yes, it updates the pip for that Python installation on your PC.

  To check which pip you're using, run:

  which pip   # or: where pip  (on Windows cmd)
  pip --version

  This tells you the path, so you can confirm if it's a venv pip or your system
  pip.

❯ also, what is the point of having a global pip? if i just use my venv?        

● Good question. If you always use virtual environments (which is the
  recommended practice), the global pip mainly serves these purposes:

  1. Creating new venvs — python -m venv .venv uses the global Python/pip as the
   base.
  2. Installing global tools — things like pipx, black, poetry, or uv that you
  want available system-wide without activating a venv.
  3. Quick one-off scripts — when you don't bother setting up a venv.

  But honestly, if you're disciplined about always using venvs, you rarely need
  to touch global pip directly. Many developers use tools like:

  - pipx — installs CLI tools globally but in isolated environments
  - uv — a faster modern replacement that handles both global tools and venvs
  cleanly

  So your instinct is correct — global pip matters less the more consistently
  you use venvs.

❯ what's this warning? can i ignore it?                                         

● Yes, you can ignore it. It's just Git telling you it will convert line
  endings:

  - LF (\n) = Unix/Mac style
  - CRLF (\r\n) = Windows style

  Your files currently use LF, but since you're on Windows, Git is converting
  them to CRLF when it checks them out. This is controlled by the core.autocrlf
  setting, which is typically set to true on Windows by default.

  It's just a warning, not an error — your files and commits will work fine.

● On Windows with bash you can also use:

  .venv/Scripts/activate && python app.py

  Or if you're in cmd/PowerShell:
  .venv\Scripts\activate && python app.py

❯ cant i do scripts/python/app.py?                                              

● Yes! You can call the venv's Python directly:

  .venv/Scripts/python app.py

  This skips activating the venv entirely and uses the venv's Python directly.

❯ is the output same?                                                           
● Yes, identical output. Running .venv/Scripts/python app.py and activating the
  venv then running python app.py both use the exact same Python interpreter and
   packages.

❯ venv/Scripts/python.exe app.py explain what this is                           

● It's a path to the Python executable inside your virtual environment, followed
   by the script to run:

  - venv/ — your virtual environment folder
  - Scripts/ — contains executables (Python, pip, etc.)
  - python.exe — the Python interpreter inside the venv
  - app.py — your Flask app to run

  So instead of using the global python, you're explicitly telling your shell to
   use the Python from your venv to run app.py.
